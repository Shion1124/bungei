[[ドメイン駆動開発設計について]]
[[ユビキタス言語を使う理由]]

ユビキタス言語を使わない場合、AIエージェントとの間で**「言葉の定義のずれ」**が生じやすく、結果として**何度も確認作業が必要になり、開発効率が著しく低下する可能性が高い**です。

ユビキタス言語は、AIとの間で**「共通言語を獲得し、認識の齟齬を最小限に抑える」**ための極めて有効な手段です。これにより、AIはより的確に意図を理解し、作業を円滑に進めることが可能になります。

ただし、それでも**最低限の確認作業は必要不可欠**です。以下にその理由と詳細を説明します。

---

### ユビキタス言語を使わない場合の問題点と確認作業の必要性

1. **定義の曖昧さと誤解:**
    
    - **問題:** 人間が使う言葉は文脈によって意味が変わったり、解釈が異なったりすることがあります。AIは文脈を完全に理解しているわけではなく、学習データに基づいた確率的な判断を行います。
        
    - **結果:** 例えば、「アクティブなユーザー」という言葉一つをとっても、人間の開発者は「過去1ヶ月以内にログインしたユーザー」を指しているかもしれませんが、AIはそのような具体的な定義を知らないと、もっと広範な意味で捉える可能性があります。
        
    - **確認作業:** AIに「アクティブなユーザーとは具体的にどのような条件を満たすユーザーですか？」と都度確認し、その定義をAIに理解させる必要があります。この確認作業は、AIへの指示自体が曖昧であるため、繰り返し発生する可能性があります。
        
2. **AIの「創造性」と「誤った推論」の危険性:**
    
    - **問題:** AIは、学習データに基づいて「それらしい」コードや処理を生成することが得意ですが、必ずしもドメインの「真実」を捉えているとは限りません。特に、定義が曖昧な場合、AIは一般的なパターンや統計的な関連性に基づいて、間違った（しかし一見もっともらしい）推論をする可能性があります。
        
    - **結果:** 例えば、AIが「キャンセル処理」に関するコードを生成した際に、実はドメインでは想定されていない「返金処理の除外」という重要なルールを見落としてしまう、といったことが起こりえます。
        
    - **確認作業:** AIが生成したコードが、ドメインの全てのルールや例外処理を網羅しているかを、人間の開発者がテストやレビューを通じて厳密にチェックする必要があります。
        
3. **指示の「複雑化」と「効率低下」:**
    
    - **問題:** ユビキタス言語で定まった概念がない場合、AIに指示を出すたびに、その都度、背景説明や定義を詳細に伝えなければ、正確なアウトプットを得られません。
        
    - **結果:** プロンプトが長くなり、作成に時間がかかります。また、複雑な指示はAIの誤解を招きやすくなり、さらに多くの確認作業を生む悪循環に陥ります。
        
4. **AIとの「共通言語の獲得」の非効率性:**
    
    - ユビキタス言語は、AIにドメインの「共通言語」を提供することで、このプロセスを効率化します。
        
    - ユビキタス言語がない場合、AIとの間に「共通言語」を構築するために、上記のような試行錯誤と確認作業を繰り返すことになりますが、それは非常に時間と労力がかかるプロセスです。AIがドメインのニュアンスを完全に理解するには、多くの例示とフィードバックが必要になるでしょう。
        

---

### ユビキタス言語によるメリットとそれでも必要な確認作業

**ユビキタス言語があれば、AIとの共通言語を獲得したことになり、作業を円滑に進めることが可能になるか？ → はい、非常に円滑になります。**

- **AIの理解度が劇的に向上:** ドメインの概念が明確な言葉で定義されているため、AIは指示されたタスクの目的、対象、必要な処理などを、より正確かつ効率的に理解できるようになります。
    
- **プロンプト作成の効率化:** 「顧客アクティベーションに必要な条件を定義してください」という代わりに、「Customer の IsActive プロパティを true に設定するための activateCustomer() メソッドを実装してください。ただし、過去30日以内に取引がない場合は false のままとしてください。」のように、ユビキタス言語と具体的なドメインルールを組み合わせた指示が可能になります。
    
- **生成コードの品質向上:** AIはドメインの意図を汲み取り、より正確で意図に沿ったコードを生成しやすくなります。
    

**それでも最低限の確認作業は必要か？ → はい、必ず必要です。**

理由は以下の通りです。

1. **AIの限界:**
    
    - AIはあくまで学習データに基づいて確率的に動作するものであり、人間のような深い洞察力や創造性、あるいは倫理的な判断を必ずしも持っているわけではありません。
        
    - ドメインの全てのニュアンス、ビジネス上の例外処理、あるいは将来的な拡張性を完璧に予測してコードを生成することは困難です。
        
2. **ドメインの進化:**
    
    - ビジネスドメインは常に変化します。ボットの運用中に新しい市場の兆候が現れたり、取引戦略の変更が必要になったりすることがあります。
        
    - AIが生成したコードが、常に最新のドメイン変化に対応できているとは限りません。
        
3. **設計意図の最終確認:**
    
    - 最終的な開発設計やコードの品質に対する責任は、開発者（プロンプトを与える人間）にあります。
        
    - AIが生成したコードが、本当にドメインの意図を正確に反映しているか、ビジネス上のリスクはないか、といった最終的な判断は、人間が行う必要があります。
        

**最低限必要な確認作業の例:**

- **コードレビュー:** AIが生成したコードが、ユビキタス言語やドメインモデルの定義と一致しているか、コーディング規約に沿っているかを確認します。
    
- **単体テスト:** 生成されたコードの各機能が期待通りに動作するかを確認するテストを実行します。
    
- **結合テスト:** 複数のコンポーネントが連携して、ワークフロー全体が正しく動作するかを確認します。
    
- **シナリオテスト:** 特定のビジネスシナリオ（例：市場暴落時の対応、API障害時の挙動など）で、ボットが意図した通りに動作するかを確認します。
    
- **ドメインエキスパートによるレビュー:** 可能であれば、ドメインの専門家（仮想通貨取引の知識を持つ人）に、生成されたボットのロジックや振る舞いがビジネス上適切であるかを確認してもらいます。
    

**結論として:**

ユビキタス言語は、AIとの開発設計プロセスにおける「共通言語」を提供し、AIの理解度と指示の精度を飛躍的に向上させるための基盤となります。これにより、確認作業の回数を大幅に減らし、開発を円滑に進めることが可能になります。しかし、AIはあくまで強力なアシスタントであり、人間による最終的な判断と確認は、高品質で信頼性の高いソフトウェアを開発するために不可欠なプロセスです。
