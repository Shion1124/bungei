## アジャイル開発 運用ルール＆テンプレート提案[[アジャイル開発（更なる推奨内容）]]

現在のプロジェクトは、ドキュメントによる作業の細分化、AIエージェントの活用、進捗の可視化など、アジャイル開発の考え方と非常に親和性の高い形で進められています。これをさらに強化し、チーム全体の生産性向上、スムーズな情報共有、効果的なイテレーションの推進を目的として、以下の運用ルールとテンプレート例を提案します。

### 提案の柱：

1. **タスク管理と進捗可視化の標準化:**
    
    - タスクの定義、粒度、ステータスの標準化
        
    - 進捗状況の可視化ツール（タスクボード）の導入・活用
        
2. **ドキュメント管理と更新サイクルの明確化:**
    
    - ドキュメント（.mdファイル）の役割と更新頻度の定義
        
    - AIエージェントとの連携によるドキュメントレビュー・改善サイクルの導入
        
3. **コミュニケーションと情報共有の促進:**
    
    - デイリースタンドアップミーティングの実施
        
    - スプリントレビューとふりかえり（レトロスペクティブ）の導入
        
4. **継続的改善の仕組み:**
    
    - 上記プラクティス自体の改善サイクル
        

---

### 運用ルール例

#### 1. タスク管理と進捗可視化

- **タスクの定義:**
    
    - 各 .md ファイル内のタスクは、特定の「ユーザーゴール」や「技術的課題」を解決するための、**実行可能な最小単位**とします。
        
    - タスクは**完了までの見積もり時間（目安）**と共に記載します。（例：[2h]、[0.5d]）
        
    - 各タスクには、担当者（AIエージェントまたは担当者名）を明記します。
        
- **ステータスの標準化:**
    
    - タスクのステータスを以下のように定義します。
        
        - TODO: 着手すべきタスク
            
        - IN PROGRESS: 現在作業中のタスク
            
        - IN REVIEW / BLOCKER: レビュー待ち、またはブロッカーが発生しているタスク
            
        - DONE: 完了したタスク
            
- **タスクボードの活用:**
    
    - GitHub Issues, Trello, Jiraなどのタスク管理ツールを利用し、タスクボード（カンバンボード）を構築します。
        
    - 各 .md ファイルで定義されたタスクをタスクボード上のカードとして登録し、ステータスをリアルタイムで更新します。
        
    - 「ToDo」「進行中」「完了」などのカラムを用意し、チーム全体で進捗状況を一覧できるようにします。
        
    - AIエージェントが作業を進める際も、タスクボード上のステータスを自動または手動で更新するようにします。
        

#### 2. ドキュメント管理と更新サイクル

- **ドキュメント（.mdファイル）の役割:**
    
    - 各 .md ファイルは、特定のフェーズや機能の開発・運用手順を定義する「作業指示書」かつ「進捗記録」として機能します。
        
    - 作業を進める中で発見された課題、新たな知見、改善点は、**即時ドキュメントに反映**させます。
        
- **更新サイクルの定義:**
    
    - **デイリー:** 作業を進める中で気づいたこと、ブロッカー発生時は即時更新。
        
    - **スプリント終了時:** そのスプリントで完了したタスクの記録、発見された課題、次回スプリントで考慮すべき点をドキュメントに追記・整理します。
        
    - **AIエージェントとの連携:** AIエージェントにドキュメントの更新・レビューを依頼する作業をルーチン化します。（例：毎日の終わりに、今日の作業内容を元に最新の .md ファイルをレビュー・更新するよう指示する）
        

#### 3. コミュニケーションと情報共有

- **デイリースタンドアップミーティング（15分以内）:**
    
    - 各メンバー（AIエージェント含む）が以下の3点を共有します。
        
        1. 昨日何をしたか
            
        2. 今日何をするか
            
        3. 何かブロッカー（障害）はあるか
            
    - AIエージェントには、チャットツール（Slackなど）への自動投稿機能や、API経由での情報連携を検討します。
        
- **スプリントレビュー:**
    
    - 各スプリント（例：1〜2週間）の終わりに、完了したタスクと成果物を共有・デモンストレーションします。
        
    - ドキュメントの内容が実際の成果と乖離していないかを確認します。
        
- **ふりかえり（レトロスペクティブ）:**
    
    - スプリントの終わりに、チームのプロセスやコラボレーションについて話し合い、改善点を見つけます。
        
    - 「うまくいったこと」「改善が必要なこと」「次に試すこと」を特定し、次のスプリントに反映させます。AIエージェントの活用方法やドキュメントの更新ルールなども、この場で議論します。
        

#### 4. 継続的改善の仕組み

- **ふりかえりの結果を次のスプリント計画に反映**させます。
    
- **AIエージェントへの指示方法や、ドキュメントの書き方自体も、チームで都度改善**していきます。

#### 3. AIエージェントへの指示フォーマット例

Generated code

```
## AIエージェントへの指示

**対象ファイル:** [@backtest_refactoring.md]([@backtest_refactoring.md](https://github.com/your_repo/blob/main/backtest_refactoring.md))

**タスク:** 【Step 2-3】クラスタリングによる自動分割の実行
**担当:** @dev_agent1
**見積もり:** [2h]

**具体的な指示:**
1.  現在の [@backtest_refactoring.md]([@backtest_refactoring.md](https://github.com/your_repo/blob/main/backtest_refactoring.md)) に基づき、Prefectのクラスタリングによる自動分割プロセスを実行してください。
2.  実行後、以下の点を確認し、結果を [@backtest_refactoring.md]([@backtest_refactoring.md](https://github.com/your_repo/blob/main/backtest_refactoring.md)) に追記・更新してください。
    *   分割されたクラスタの数と各クラスタのデータ量。
    *   分割プロセス自体の実行時間と成功/失敗ステータス。
    *   分割アルゴリズムのパラメータとその決定理由。
3.  もし実行中に予期せぬエラーが発生した場合、エラーメッセージと共にブロッカーとしてタスクボードに報告してください。

**完了報告:** 完了後、タスクボードのステータスを「IN PROGRESS」から「DONE」へ更新してください。
```

これらの運用ルールとテンプレートを導入することで、チーム全体の開発プロセスがより明確になり、AIエージェントとの連携もさらにスムーズになるはずです。プロジェクトの進行状況やチームの規模に合わせて、これらの要素を適宜調整してください。
