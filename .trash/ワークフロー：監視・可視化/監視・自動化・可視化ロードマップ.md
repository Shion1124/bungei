### 【第1段階】Prefectフロー堅牢化・自動再実行

- PrefectのTask分割（データ取得→特徴量生成→クラスタリング→バリデーション→出力）
- 各Taskでtry/except＋Prefectの`fail`/`retry`を活用
- Prefectフロー定義で`retries`/`retry_delay_seconds`を設定
- 失敗時は詳細なエラーログを出力（どのTask・どのデータで失敗したか明示）
- Prefect UI/CLIでリトライ・失敗時の挙動を検証

### 作業実行：
- PYTHONPATH=. uv run python scripts/ml_pipeline.py
- prefect flow-run ls（CLIで確認）
- prefect flow-run inspect <FLOW_RUN_ID>
- ### 実行ログ・状態から分かること
	- uv run python scripts/ml_pipeline.py でPrefectフローが正常に実行され、
		- データ取得
		- 特徴量生成
		- クラスタリング
		- バリデーション
	- 各Taskがすべて「Completed」状態で完了しています。
	- Prefectのリトライ・エラーハンドリング・詳細ログ出力も有効です。
	- CLIのprefect flow-run lsでも「main-flow」が「COMPLETED」となっており、過去の失敗・成功履歴も管理されています。
	- UIにも履歴・ログ・リトライ情報が記録されているはずです。

PrefectSlack通知・エスカレーション設計

通知内容にはフロー名・Task名・エラー内容・実行ID・Prefect UIリンク・発生時刻を含める  
Webhook URLはチャンネルごとに分けて管理

通知ポリシー例  
#alerts-trading: Run states = Failed, Cancelled  
#alerts-critical: Run states = Crashed, RetryFailed（またはFailedも含める）

運用上の注意点  
通知が多すぎる場合はRun statesを絞る（All run statesは非推奨）  
通知が届かない場合はWebhook URL・Prefect UI設定・Slack側の権限を確認  
Prefectのバージョンによっては「Notifications」→「Automations」への移行も検討

トラブルシュート例  
通知がSlackに届かない→Webhook URL/チャンネル権限/Prefect UI設定を再確認  
通知が多すぎる→通知ポリシーでRun statesを絞る  
Prefect UIで通知がONになっているか確認

## 4. 考慮すべき点・注意点

- **再現性・環境依存排除**: uv＋PEP723運用、パス・依存管理の徹底
- **エラーハンドリング・ログ**: 失敗時の原因特定が容易な設計
- **通知ノイズ抑制**: 本当に必要な時だけ通知・エスカレーション
- **監視粒度**: フロー全体だけでなくTask単位の監視も検討
- **スケーラビリティ**: フロー・Taskの増加やデータ量増大に耐えうる設計
- **保守性**: 設定・通知先・パラメータの外部化、ドキュメント整備
- **セキュリティ**: WebhookやAPIキーの管理、監視情報のアクセス制御
- **現場運用性**: Streamlit等のUIは現場の声を反映し、使いやすさを重視
